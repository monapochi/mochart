# 【開発エピソード】Mochart WebGPU版：「ゲーム品質モーションブラー」への7つのフェーズと、華々しい失敗の記録

本ドキュメントは、2026-03-03のMochartペアプログラミング・セッションにおける `experiment/motion-blur` ブランチの記録です。「60fps超でヌルヌル動かしたい」というシンプルな欲望から始まった実験は、7フェーズにわたる試行錯誤の末に「チェックしても外しても同じ」という残酷な結末を迎えました。失敗もまたエンジニアリングの資産、として技術記事ベース用に記録します。

---

## エピソード1：動機――「どうせなら映画みたいに動かせないか？」
**〜 VRAMダイエット成功の余韻の中で生まれた欲 〜**

ExecutionPlan DSLによるVRAM 10KB未満という成果が出た直後、「せっかくWebGPUがあるんだから、エフェクトも試したい」という気持ちが湧き上がりました。

ターゲットは**モーションブラー**。株チャートをパンすると、ローソク足がカクカクではなく映画フィルムのようにブレながらスライドする体験です。`experiment/motion-blur` ブランチを `feat/execution-plan-engine` から切り、実験開始。

---

## エピソード2：テクスチャペアの導入と「歴史バッファ」
**〜 フレームN と フレームN-1 を持てば何でもできる、はずだった 〜**

モーションブラーの基本は「現在フレームと前フレームを合成すること」。
そのために2枚のオフスクリーンテクスチャを確保しました。

- `_mbOffscreen` ： 現在フレームの描画先
- `_mbHistory`   ： 前フレームの保存バッファ

実装上の最難関は**「history commit のタイミング」**でした。

最初の実装では、`drawGpu()` の末尾でオフスクリーンを history にコピーしていました。しかし、これだとフレーム補完ループ（後述）が走っている間に history が上書きされ、正しい「N-1フレーム」が消えてしまいます。

正解は **`drawGpu()` の先頭（offscreen 上書き前）に `_commitHistory()` を呼ぶ** こと。この1行の移動で、フレーム間ループ全体が安定しました。

---

## エピソード3：フレーム補完ループの誕生
**〜 60fps でも描画は最大 16ms に 1 回 ── その間を埋めたい 〜**

WebGPUレンダラーの`Atomics.waitAsync`ベースのrenderループは、データ更新時のみ起床します。つまりパン中も実際の再描画は**最大60fps（16ms間隔）**。1フレーム間の移動量が大きいと、人間の目にはカクカクと映ります。

解決策として `render_worker.js` に**インターフレームタイマー**を実装しました。

```javascript
// 実フレーム描画後、~16ms かけて uvOffset を 0 に向けて減衰させる
_startInterpLoop() → _scheduleInterpTick() → setTimeout(4ms) × n
```

原理としては：
1. 実フレームで「前フレームからの UV シフト量（`_mbCurUvOff`）」を記録
2. タイマーで 4ms ごとに `_blitXxx(uvOff * s, alpha * s)` を呼ぶ
3. `s` を 1→0 に減衰させ、フレームが「元の位置にスライドイン」する様子を演出

## エピソード4：7フェーズの試行錯誤
**〜 SimpleBlend → VelocityGated → HalfStep → FullShift → PositionInterp → DuplicateFix → N-tap 〜**

| フェーズ | 内容 | 結果 |
|---|---|---|
| 1. SimpleBlend | `mix(N, N-1, alpha)` 2-tap lerp | ブラーが常にかかり止まらない |
| 2. VelocityGated | 移動量 EWMA でアルファを0-gating | 静止時消えるが動き中は ghosty |
| 3. HalfStep | UV を半分にずらして補間 | 微妙な改善、まだ不自然 |
| 4. FullShift | delta bars 分フル UV シフト | ズームイン時に二重フレーム出現 |
| 5. DuplicateFix | 30px クランプで二重フレーム解消 | ズームイン大移動時の blur が弱くなる |
| 6. N-tap VelocityBlur | 独立2テクスチャでクランプ不要に、smooth-step 8-tap | 理論上は正しいはず…… |
| 7. ZoomIn Adaptive | ease-out cubic + adaptive duration 16–40ms | 「まだカクカク」…… |

## エピソード5：N-tapシェーダーの設計ミスと気づき
**〜 Prev テクスチャを「シフトした UV」でサンプリングする罠 〜**

フェーズ6で実装した `effect_velocity_blur.wgsl` は、理論的にはゲーム品質のはずでした。

```wgsl
// ❌ 旧シェーダーの問題
curr = frame_tex @ uv           // バーXの色
prev = prev_tex  @ (uv + vel)   // ← バーXの色（パン後に同一内容が vel 分移動している）
mix(curr, prev, t)              // → 同じ色同士のブレンド → 視覚変化ゼロ
```

チャートはパンすると**全バーが vel だけ移動**する。つまり `prev @ (uv + vel)` は「同じバーが1フレーム前にいた位置」であり、色は `curr @ uv` と**ほぼ同一**。ブレンドしても何も変わりません。

「効果がわからない」という報告を受けて原因を診断し、正しいアプローチに気づきました。

```wgsl
// ✅ 正しいアプローチ：現在フレームをモーションベクトル方向に N-tap サンプリング
for i in 0..samples:
    trail_x = uv.x + vel_uv_x * smoothstep(t)
    accum += frame_tex @ (trail_x, uv.y) * (1 - smoothstep(t))
// バーエッジのピクセルは「バー内←→背景」の間をサンプリング → スミアが生まれる
```

バーのエッジ周辺では、trail direction に沿ったサンプルが「バー内側」と「背景/隣バー」の両方から拾うことで、自然な方向性ブラーが生まれる──**はずでした**。

こちらも修正後「チェックを入れても外しても変わらない」という評価となり，実験終了となりました。

## エピソード6（エピローグ）：なぜ失敗したのか？ ── WebGPUとチャート描画の構造的相性問題
**〜 "正しいアルゴリズム" が "正しい体験" にならない理由 〜**

失敗の根本原因を振り返ると、いくつかの構造的な問題が見えてきます。

**1. チャートはピクセルの移動ではなく、バーの再描画**

ゲームのモーションブラーが機能するのは、**オブジェクトが画面空間を移動する**からです。背景は静止し、キャラクターだけが動く。その「動いたピクセル」にブラーをかけると自然に見えます。

株チャートのパンは違います。画面上のすべてのバーが右から左へ移動するのではなく、**バッファをシフトして全バーを描き直す**。つまり「画面上のある位置にいたピクセル」が「別の位置に移動する」のではなく、その位置に「別のバーが描画される」だけです。N-tap も ghosting も、この構造では意味をなしません。

**2. 真に必要なのは velocity buffer**

正しいゲーム品質モーションブラーには、各ピクセルの「速度ベクトル（velocity buffer）」が必要です。これはジオメトリの「前フレームの位置」と「現在フレームの位置」の差分をGPUで計算し、それを使ってブラー方向を決定するものです。チャートでこれを実現するには、全バーの「前フレームにおけるスクリーン座標」をGPUに精確に渡す必要があります。現在のアーキテクチャではその情報は保持していません。

**3. 知覚的な要因**

ローソク足チャートは縦の帯が並ぶ、高コントラストの規則的なパターンです。隣接するバーは同色に近いことも多く、「スミア」が生じていても視覚的に識別しにくい。同じアルゴリズムでも、動くキャラクターや複雑なシーンなら劇的に見えたかもしれません。

## 学びのまとめ

- **ドメインの物理モデルを正しく理解する**：「画面上のピクセルが動く」のか「別のコンテンツが描画される」のかで、blur の理論が根本的に変わる。
- **アルゴリズムの正しさと知覚的な効果は別物**：正しい N-tap も、ドメイン特性が合わなければ効果がゼロになりうる。
- **失敗のブランチは消さない**：`experiment/motion-blur` には7フェーズの試行が刻まれている。「なぜ動かないのか」を理解した今、将来 velocity buffer 方式で再挑戦するときの出発点になる。
- **ポストモーテムを書く**：「動かなかった」で終わらせず、なぜ動かなかったかを記録することが次の設計品質を引き上げる。
