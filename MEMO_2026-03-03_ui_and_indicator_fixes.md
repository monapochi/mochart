# 【開発エピソード】Mochart WebGPU/WASM版：UIの小さな洗練からRustコアエンジンのバグハントへ

本ドキュメントは、2026-03-03に行われたMochartのペアプログラミング・セッションの記録です。「クロスヘアのポップアップ表示を最適化する」という小さなUI改善から始まり、最終的にRust/WASMエンジンのコアロジックの大規模な改修に至るまでの軌跡を、技術記事（Zenn/Qiita等）の執筆ベースとして利用できるようにエピソード単位で詳細に記録しています。

---

## エピソード1：発端は「ホバーした時だけ表示したい」という小さな要望から
**〜 DPR（デバイスピクセル比）がもたらす座標系の罠 〜**

すべては「常に表示されているクロスヘアのデータポップアップを、ローソク足や線にホバーした時だけ表示するようにしたい」という、ちょっとしたUIのブラッシュアップ要請から始まりました。

まずは `render_worker.js` 内に、マウスのY座標を価格に逆算し、表示されているローソク足や各種インジケーター（SMAなど）の価格と照らし合わせる簡易的なヒットテストロジックを実装しました。ところが、実際にブラウザで確認してみると、**「マウスカーソルの位置と、実際にヒット判定されるエリアが上下にズレている」**という現象が発生しました。

原因は、描画層ごとの座標系の違いでした。
バックグラウンドで走るWebGPUのレイアウト（`gpuRenderer.paneLayout`）は**物理ピクセル（Canvas backing-storeの解像度）**で精緻に計算されています。一方で、前面のCanvas 2Dコンテキスト（HUD層）やマウスイベントは**CSS論理ピクセル**の世界で動いています。
ヒットテストを行う際、このミスマッチを考慮せず物理ピクセルの値をそのまま使ってしまったため、高解像度ディスプレイ（DPR > 1）の環境下でズレが生じていたのです。HUD側にレイアウト境界を引き渡す直前で `DPR (devicePixelRatio)` で割る補正処理を入れ、このズレを完璧に解消しました。

## エピソード2：「GC対象増えてませんか？」—— ユーザーの鋭い指摘
**〜 毎フレーム処理におけるゼロアロケーション（Zero Allocation）の徹底 〜**

ホバー判定が上手く動くようになり一息ついた直後、ユーザーから非常に鋭い指摘が飛びました。
*「マウスを動かすたびにオブジェクトが生成されて、GC（ガベージコレクション）の負担になっていませんか？」*

確認してみると、ホバー情報をHUDに渡すために、毎フレーム `const popupData = { open: ..., high: ... }` のように新しいオブジェクトを生成しては破棄していました。60fpsでマウスをグリグリ動かす環境において、これは「チリツモ」でパフォーマンス低下やGCスパイク（画面の引っかかり）を引き起こす典型的なアンチパターンです。

そこで即座に**「ゼロアロケーション」**へのリファクタリングを決行しました。
ファイルのトップレベルで `const _popupData = { ... }` という単一のオブジェクト（使い回し用のプール）を初期化しておき、ループの中ではそのプロパティの「数値（プリミティブ値）」だけを上書き更新するように変更しました。DPR補正用のレイアウトオブジェクト新規生成もあわせて撤廃し、描画のホットパスでのメモリ確保を完全にゼロに抑え込みました。

## エピソード3：画面リロード直後の「幽霊（ゴースト）」退治
**〜 SharedArrayBufferとアトミック操作の初期値問題 〜**

次なる不具合は、画面をリロードした瞬間に現れました。まだマウスを一ミリも動かしていないのに、画面の左上端（座標: 0, 0）にクロスヘアやポップアップデータが「フライング」で描画されてしまっていたのです。

Mochartでは、メインスレッドと描画用Worker間の高速通信に `SharedArrayBuffer` (SAB) と `Float32Array/Int32Array` を用いています。問題は、このメモリ領域を確保した際の初期値が「0」であることでした。Worker側は律儀に「ポインターの初期位置は X:0, Y:0 だな」と解釈して描画を走らせていたわけです。

解決策として、メインスレッド（`chart_host.js`）の起動直後に、ポインター座標用のSABスロットへ明示的に `f32ToI32(-1.0)` を叩き込むように修正しました。これにより、初期状態は「画面外にマウスがある」と認識され、不気味なゴースト表示は見事に消滅しました。

## エピソード4：レイアウトのせめぎ合い —— サブペイン vs 日付軸
**〜 描画領域の物理マージン調整 〜**

UI関連の最後のバグは「Volumeなどのサブペインを表示した際、画面の一番下に描かれる日付軸のテキストとローソク足が重なって見づらい」というものでした。

先ほどのDPRの知識を活かし、WebGPU側のレイアウト計算に手を入れることにしました。`gpu_renderer.js` でペイン分割を行う際、下端までぎっしり使い切るのではなく、「常にボトムに8px（論理ピクセル）のマージンを残す」ように計算式を修正。内部的には `8 * DPR` として物理ピクセルに変換して引き算を行いました。
これにより、チャート全体がスッと上に押し上げられ、日付軸の視認性がクリアに確保されました。

## エピソード5：最大の難関「遡ると線が切れる！」
**〜 Rust/WASMエンジンへのダイビング 〜**

ここまではJS層でのUI調整でしたが、過去データに向かって勢いよくチャートをスクロールしていたところ、**「SMAやボリンジャーバンドなどの線が、データの左端（最古のデータ）付近にくると急にプツンと途切れてしまう」**という致命的なバグが発覚しました。

いよいよRustで書かれたデータエンジンの深部、`execution_plan.rs` への突入です。
インジケーターの計算には、現在画面に見えている範囲のデータに加えて、「過去の履歴データ（`pre_history`）」が必要です。（例：SMA(20)なら直近19本のデータ）。

コードの該当箇所を解析した結果、当時のロジックは非常に「大雑把かつ豪快」な仕様になっていました。
*「要求された過去履歴（`pre_need`）が完全に確保できない場合（＝左端ギリギリまで来て足りなくなった場合）、**今持っている履歴データをすべて破棄してしまい、強制的に計算不能なNaN（Warmup）の期間として扱う**」*
この仕様のせいで、本来ならまだ計算・描画できるはずの部分まで余分に削り取られていたことが判明したのです。

## エピソード6：限界までデータを絞り出せ！
**〜 `saturating_sub` と動的Warmupの恩恵 〜**

この問題を根本から解決するため、Rust側のデータフェッチ戦略を大きく再構築しました。
必要な履歴が完全に足りていなくても**「いま存在するだけの Partial（部分的）な履歴データを可能な限りかき集めて投入する」**ように `prepare_close_input` メソッドを改修。

さらに、各インジケーター（SMA、EMA、MACD、ボリンジャーバンドなど）の出力側オフセット処理も書き換えました。単純な固定期間を切り捨てるのではなく、Rustの `saturating_sub` を活用し、「本当に足りていないデータ分」だけを動的に算出して、正確な値だけをWarmup期間（NaN）としてマッピングするように数学的な境界（バウンダリ）管理を厳格化しました。

この改修によって、インジケーターの線は過去データの左端ギリギリまで、途切れることなく美しく伸びるようになりました。

## エピソード7（エピローグ）：思わぬ副産物とパフォーマンスの飛躍

Rust側のバウンダリ管理を徹底的に見直した結果、単に線の途切れバグが直っただけでなく、**思わぬ副産物**が得られました。

「必要な箇所だけを的確に計算する」というアーキテクチャがより強固になったことで、ウィンドウのリサイズ時や、ドラッグ・拡縮によって表示されるバー数（`visBars`）が変動した際でも、**不要なRust側エンジンの再コンパイル・再計算が走らなくなりました。**

UIの小さな「ホバー判定」をきっかけに進んだバグハントの旅は、最終的にコアエンジンのメモリ管理と計算効率を飛躍的に高める結果となり、画面のレスポンスと描画パフォーマンスのさらなる向上をもたらしました。

---
**学びのまとめ：**
- **描画環境の罠**：WebGPU（物理）とCanvas 2D/マウス（論理）のピクセル系は常に意識すべし。
- **ホットパスの原則**：60fpsの世界では、細かいオブジェクト生成が命取り。使い回し（Zero Allocation）を徹底すべし。
- **エッジケースの追求**：配列の境界（左端・右端）での挙動の妥協は、UI上の違和感としてすべて跳ね返ってくる。限界までデータを有効活用する境界設計が、結果的にシステム全体の安定性と速度を生む。
